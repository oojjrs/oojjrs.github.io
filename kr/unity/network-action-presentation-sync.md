네트워크 게임의 연출은 Animator 상태를 직접 동기화하기보다, 서버와 클라이언트가 같은 액션 의미와 순서를 공유하고 각 클라이언트가 그 상태를 로컬 연출로 풀어내는 편이 안정적이다.
{: .article-lead }

> 네트워크는 동작의 의미와 순서를 맞추고, 로컬은 그 상태를 기준으로 연출을 재생하거나 폐기한다.
{: .article-principle }

도식 기준: 본선은 명령/의도 생성 -> 액션/상태 요청 -> 판정 주체 -> 공유 액션 상태 -> 연출 어댑터 -> 출력으로 이어지는 직선 흐름이다. 액션 규칙 정의와 연출 정의는 본선을 직접 대체하지 않고, 판정 주체와 연출 어댑터가 참조하는 정적 정의다.
{: .article-note }

시작, 중단, 대체 의도는 모두 같은 명령 생성 단계에서 생길 수 있고, 액션 요청 역시 클라이언트, 서버, AI, 로컬 시스템 어디서든 만들어질 수 있다. 동시에 중단이나 대체는 요청으로만 들어오는 것이 아니라, 판정 주체가 우선순위·피격·사망·상태 규칙을 처리한 결과로 결정될 수도 있다. 어느 쪽이든 기존 액션을 끊거나 바꾸는 최종 결정은 판정 주체 쪽에서 확정하고, 그 결과는 공유 액션 상태에 기록되어 동기화된다. 연출 어댑터는 판정 주체에게 직접 호출받지 않고, 공유 액션 상태 변경을 감지해 애니메이션·이펙트·사운드를 갱신한다.
{: .article-note }

## 문제 정의 {#problem}

Unity Animator는 표현을 만들기에는 좋지만, 네트워크 동기화의 기준으로 삼기에는 불안정하다. Animator Controller, transition, clip 길이, layer, animation event, Unity 버전 차이가 섞이면 같은 명령을 받아도 클라이언트마다 재생 상태가 달라질 수 있다.

따라서 동기화 기준은 Animator 내부 상태가 아니라 게임이 이해하는 액션 상태여야 한다. 예를 들면 "지금 어떤 동작을 시작했는가", "그 동작의 순번은 무엇인가", "목표와 방향은 무엇인가", "연출을 새로 시작해야 하는가" 같은 정보다.

## 동기화 방식 {#sync-model}

Unity의 기본 네트워크 컴포넌트가 제공하는 방식은 대체로 state replication 또는 snapshot replication에 가깝다. 위치, 회전, Animator parameter, 재생 상태처럼 이미 만들어진 결과 상태를 계속 복제하고, 받는 쪽은 그 결과를 따라간다. 직관적이고 만들기 쉽지만, 동기화해야 할 결과물이 많아질수록 전송량과 보정 비용이 커진다.

이 구조는 그보다 command/result-based synchronization에 가깝다. 모든 중간 좌표나 Animator 진행률을 맞추려 하지 않고, 명령·요청과 판정 주체가 확정한 액션 결과만 공유한다. 네트워크 상태는 항상 균일하지 않고 지연, 순서 뒤바뀜, 일시적인 끊김이 생길 수 있다고 전제한다. 그래서 연속적인 표현을 매 순간 맞추기보다, 액션의 의미와 전환점만 동기화하고 애니메이션, 이펙트, 사운드는 각 클라이언트가 로컬에서 재구성한다.

이 방식은 deterministic lockstep과도 다르다. deterministic lockstep은 같은 입력을 같은 tick과 같은 순서로 처리해 모든 클라이언트가 중간 상태까지 같은 결과를 내야 한다. 여기서는 그런 완전 결정적 시뮬레이션을 필수 조건으로 두지 않는다. 필요한 것은 모든 클라이언트가 같은 판정 결과와 액션 상태 전환을 공유하고, 각자 그 상태를 자기 로컬 연출로 풀어내는 것이다.

## 공통 구조 {#structure}

이 구조는 물리적인 실행 계층이라기보다 세 책임 영역으로 나뉜다.

- 요청 생성 계층: 클라이언트 입력, 서버 스크립트, AI, 로컬 시스템이 시작·중단·대체 의도를 같은 단계에서 만들 수 있다.
- 판정/동기화 계층: 들어온 시작·중단·대체 요청을 검증하고, 우선순위·피격·사망·상태 규칙에 따라 중단이나 대체를 자체 결정할 수도 있다. 최종적으로 가능 여부, 취소·연계·피격 우선순위, 판정 타이밍을 확정하고, 그 결과를 공유 액션 상태로 동기화한다.
- 로컬 연출 계층: 판정 주체에게 직접 호출받지 않고 공유 액션 상태의 변경을 감지한 뒤, 애니메이션·이펙트·사운드·투사체·판정을 재생하거나 정리한다.

직선 흐름을 유지하려면 핵심 데이터는 작아야 한다. 액션 상태에는 현재 동작, 일련번호나 버전, 목표, 위치, 시작 시각, 중단·대체 결과 정도만 들어간다. 반대로 연출 세부사항은 액션 정의와 연출 묶음으로 분리한다.

```text
액션 정의
- 애니메이션 매핑
- 지속 시간 / 충돌 시간 / 연계 시간
- 취소 정책
- 다음 동작 규칙

연출 묶음
- 부착 지점
- 지연 시간
- 이펙트 / 사운드
- 투사체 / 판정 / 표식
```

이렇게 나누면 네트워크는 작은 상태만 공유하고, 클라이언트는 자기 환경에 맞는 Animator와 연출 자산을 사용할 수 있다.

## 판정 주체의 위치 {#decision-owner-placement}

시작, 중단, 대체 의도는 모두 같은 요청 생성 단계에서 생길 수 있다. 동시에 중단과 대체는 판정 주체가 기존 액션의 상태와 새 사건의 우선순위를 비교한 결과로 결정될 수도 있다. 여기서 위치를 분명히 해야 하는 것은 최종 판정 주체다. 핵심은 "누가 현재 액션을 최종 판정하고, 그 결과를 다른 계층이 따르게 만드는가"다. 게임 구조에 따라 그 판정 위치는 달라진다.

| 게임 형태 | 판정 주체의 실제 위치 | 설명 |
| --- | --- | --- |
| 네트워크 없는 게임 | 로컬 런타임 객체 | 특정 namespace나 scene 안의 MonoBehaviour, service, manager가 액션 가능 여부와 취소 규칙을 판정한다. 네트워크 전송은 없지만 구조적으로는 같은 역할을 한다. |
| 로컬 협동 / 싱글 시뮬레이션 | 대표 로컬 시뮬레이션 | 여러 입력 소스가 있어도 하나의 로컬 gameplay runtime이 최종 액션 상태를 만든다. |
| P2P / Host 방식 | Host 또는 Peer Master | 통신은 peer 사이에서 일어나더라도, 한 peer가 master 역할로 세션의 기준 상태를 판정한다. 이 경우 "소켓 서버"라기보다 Host 또는 Peer Master라고 부르는 편이 역할이 선명하다. |
| Client-Server 구조 | 게임 서버 / dedicated server | 클라이언트는 요청을 보내고, 서버가 액션 가능 여부와 타이밍을 확정한 뒤 공유 상태를 배포한다. |

따라서 이 구조는 네트워크 전용 패턴이 아니다. 오프라인 게임에서는 같은 개념이 로컬 코드 안에 접혀 있고, 네트워크 게임에서는 최종 판정 위치가 Host나 서버로 이동한다. 달라지는 것은 판정 위치이지, "시작·중단·대체 의도나 사건을 만드는 쪽"과 "액션을 최종 판정하는 쪽", 그리고 "동기화된 액션 상태를 연출로 푸는 쪽"을 분리한다는 원칙은 같다.

## 장점 {#strengths}

가장 큰 장점은 동기화 단위가 명확하다는 점이다. Animator trigger나 transition이 아니라 액션의 의미와 순서를 맞추기 때문에 Unity 버전이나 프레임 차이에 덜 흔들린다.

연출 교체도 쉬워진다. 같은 액션 상태에 대해 캐릭터별 애니메이션, 이펙트, 사운드, 히트박스 표현을 다르게 붙일 수 있다. 판정 주체나 액션 규칙을 바꾸지 않고 표현만 바꾸는 일이 가능해진다.

취소와 연계도 시스템화하기 좋다. 현재 액션을 즉시 끊을지, 입력을 예약할지, 충돌 이후에만 다음 액션을 허용할지 같은 규칙을 한 곳에서 판단할 수 있다.

마지막으로 디버깅이 쉬워진다. 문제가 생겼을 때 "공유 액션 상태가 잘못됐는가", "연출 어댑터가 잘못 해석했는가", "출력 자산이 잘못 붙었는가"를 나눠 볼 수 있다.

## 단점과 위험 {#risks}

초기 구조는 단순 Animator trigger 방식보다 무겁다. 액션 정의, 연출 묶음, 취소 정책, 일련번호, 연출 정리 규칙이 필요하기 때문이다. 작은 오프라인 게임에는 과한 설계일 수 있다.

타이밍 데이터 관리도 까다롭다. 게임플레이 판정 시간과 연출 시간이 어긋나면 화면에서는 맞은 것처럼 보이는데 실제 판정은 아직이거나, 이미 피해가 적용됐는데 이펙트가 늦게 나오는 문제가 생긴다.

이 부담은 제작 파이프라인으로 어느 정도 줄일 수 있다. animation clip에서 길이, 이벤트 위치, 충돌 프레임, 취소 가능 구간, 이펙트·사운드 발화 시점 같은 메타데이터를 추출해 액션 데이터의 기본값으로 만들면 수작업 부담과 데이터 불일치를 줄일 수 있다. 다만 clip은 동기화의 기준이 아니라 액션 정의를 작성하기 위한 원천 자료에 가깝고, 최종 판정 규칙과 예외 처리는 액션 시스템 쪽에서 명시적으로 관리해야 한다.

복잡도가 한 곳으로 몰릴 위험도 있다. 연출 어댑터가 Animator, 이펙트, 사운드, 투사체, 판정을 모두 받아주기 때문에 경계를 세우지 않으면 거대한 중재자가 된다.

취소 정책은 특히 조심해야 한다. "즉시 취소", "예약", "충돌 이후 취소", "취소 불가" 같은 규칙이 많아지면 디자이너와 프로그래머 모두 현재 상태를 예측하기 어려워진다.

P2P/Host 방식에서는 Host 또는 Peer Master에 부담이 집중된다. Host는 자기 로컬 플레이를 처리하면서 다른 peer의 요청을 검증하고, 액션 결과를 확정하고, 공유 상태를 다시 배포해야 한다. 구조 전체의 전송량은 줄어들 수 있지만 Host의 네트워크 품질, 업로드 대역폭, 처리 성능이 세션 전체의 안정성에 직접 영향을 준다.

PvP 게임에서는 판정 주체의 신뢰성이 더 큰 문제가 된다. 플레이어 중 한 명이 Host 또는 Peer Master라면 그 peer가 액션 가능 여부, 피격, 취소, 타이밍을 확정하는 심판 역할까지 맡게 된다. 이 경우 구조는 가볍지만 핵이나 조작, host advantage에 취약해질 수 있다. PvP에서는 가능한 한 dedicated server나 제3의 판정 서버(referee server)처럼 신뢰 가능한 판정 주체를 두는 편이 안전하며, 단순 relay 서버는 패킷 전달만 할 뿐 판정을 검증하지 않으므로 같은 보안 대책이 되지 않는다.

## 적용 원칙 {#principles}

좋은 균형은 다음 원칙에서 나온다.

- Animator는 동기화 기준이 아니라 출력 장치로 둔다.
- 네트워크는 액션의 의미, 순서, 목표, 시작 시각, 판정 결과만 공유한다.
- 액션 정의는 작게 유지하고, 연출 묶음은 별도로 둔다.
- 취소 정책은 암묵적 코드 흐름이 아니라 명시적인 규칙으로 둔다.
- 새 액션이 시작되면 이전 액션의 예약 연출과 추적 이펙트를 정리한다.
- 게임플레이 판정 시간과 연출 시간의 관계를 문서화한다.
- animation clip에서 추출한 타이밍 메타데이터는 기본값으로만 쓰고, 최종 규칙은 액션 정의에서 명시한다.
- PvP에서는 플레이어 Host보다 dedicated server나 referee server 같은 신뢰 가능한 판정 주체를 우선 검토한다.

이 구조의 목적은 Animator를 배제하는 것이 아니다. Animator가 가장 잘하는 표현 재생은 그대로 맡기되, 네트워크 게임의 진실은 더 작고 명확한 액션 상태에 두는 것이다.
