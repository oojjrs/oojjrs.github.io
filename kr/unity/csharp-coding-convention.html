Unity C# 코딩 컨벤션

Home
목적
  • 코드 모양을 고민하는 시간을 0에 가깝게 줄인다.
  • 도메인 숙련도가 높아질수록 코드 분석 속도가 0에 가까워지게 한다.
수단
  • IDE와 표준 도구의 기본 규칙을 최대한 그대로 쓴다.
  • 수동 코딩 중 판단 기준을 단순하게 유지한다.
  • 동사-함수, 명사-변수처럼 컨텍스트 정보를 일관되게 둔다.
  • 최종 검수와 수동 코딩 기준을 맞추기 위해 AI에게도 동일하게 요구한다.
서드파티 코드 영역은 이 컨벤션 적용 대상이 아니다. 기존 도메인이 존재하는 곳은 항상 기존 원칙을 우선한다.

이 문서에서 언급하지 않은 컨벤션은 Unity 기준과 Visual Studio Code Style 기본 옵션을 따른다.

관련 필수 지침

로그를 추가하거나 수정할 때는 종류, 대문자 문구, identifier 참조, 안정성 규칙을 정한 로그 작성 지침을 함께 따른다.

명명파일과 Unity 객체단어구문과 구성
SECTION

명명

01

인터페이스는 Interface 접미사를 쓴다

올바른 사용
public interface FunctionInterface
{
}

private FunctionInterface function;
잘못된 사용
public interface IFunction
{
}

private FunctionInterface functionInterface;

Interface 표시는 타입명에만 둔다.

02

static 멤버 변수는 밑줄 2개를 붙인다

올바른 사용
public static int __Index;
private static int __index;
잘못된 사용
public static int Index;
private static int _index;
private static int index;

public static은 __PascalCase, private static은 __camelCase로 둔다.

03

const와 readonly는 PascalCase를 쓴다

올바른 사용
private const int MaxCount = 10;
private readonly int StartIndex;
잘못된 사용
private const int max_count = 10;
private readonly int start_index;

상수형 멤버도 타입/속성 이름처럼 읽히게 한다.

04

속성은 명사 또는 형용사로 쓴다

올바른 사용
public bool IsActive { get; }
public bool HasReward { get; }
public Item AsItem { get; }
잘못된 사용
public bool Active { get; }
public bool Activate { get; }
public bool Reward { get; }

묻는 형태의 bool 속성은 Is, Has 같은 접두사로 IDE 검색을 모은다.

05

Can과 -able의 의미를 구분한다

올바른 사용
public bool CanGather { get; }
public bool Gatherable { get; }
잘못된 사용
public bool CanGathered { get; }
public bool Gather { get; }

CanGather는 객체가 하는 행위, Gatherable은 객체가 가진 속성이다.

06

검색성이 중요한 속성에는 타입 postfix를 붙인다

올바른 사용
public Sprite IconSprite { get; }
public GameObject HitEffect { get; }
public string DescriptionTooltip { get; }
잘못된 사용
public Sprite Icon { get; }
public GameObject Hit { get; }
public string Description { get; }

String, Sprite, Prefab, Effect, Tooltip처럼 검색 필터가 필요한 경우만 쓴다.

07

함수는 동사 또는 동사구로 쓴다

올바른 사용
public void GiveReward()
{
}
잘못된 사용
public void Reward()
{
}

동작은 함수명에서 바로 드러나야 한다.

08

Enum 타입은 Enum 접미사를 쓴다

올바른 사용
public enum ItemGroupEnum
{
}

private ItemGroupEnum itemGroup;
잘못된 사용
public enum ItemGroup
{
}

private ItemGroup itemGroupEnum;

변수명에는 Enum을 반복하지 않는다.

09

코루틴 함수는 Coroutine 접미사를 쓴다

올바른 사용
private IEnumerator FadeCoroutine()
{
    yield return null;
}
잘못된 사용
private IEnumerator Fade()
{
    yield return null;
}

실행 방식이 다른 함수를 이름에서 구분한다.

SECTION

파일과 Unity 객체

10

파일명과 대표 정의명을 맞춘다

올바른 사용
// File: PlayerController.cs
public sealed class PlayerController
{
}
잘못된 사용
// File: Player.cs
public sealed class PlayerController
{
}

IDE와 IntelliSense에서 파일과 타입을 바로 찾을 수 있어야 한다.

11

파일 1개에는 대표 정의 1개만 둔다

올바른 사용
// File: PlayerController.cs
public sealed class PlayerController
{
    private enum StateEnum
    {
    }
}
잘못된 사용
// File: PlayerController.cs
public sealed class PlayerController
{
}

public struct PlayerState
{
}

public enum PlayerModeEnum
{
}

class, struct, enum 등 대표 정의는 파일을 나눈다. 중첩 정의는 예외다.

12

중첩 타입은 enum, struct, interface, class 순서로 둔다

올바른 사용
public sealed class PlayerController
{
    private enum StateEnum
    {
    }

    private struct Entry
    {
    }

    private interface CacheInterface
    {
    }

    private sealed class Cache
    {
    }
}
잘못된 사용
public sealed class PlayerController
{
    private sealed class Cache
    {
    }

    private interface CacheInterface
    {
    }

    private enum StateEnum
    {
    }

    private struct Entry
    {
    }
}

한 파일의 여러 타입 선언은 대표 정의 1개 원칙 때문에 보통 중첩 타입에만 허용한다. 중첩 타입 선언 순서는 enum > struct > interface > class다.

13

파일명과 클래스명은 명사로 둔다

올바른 사용
// File: PlayerMover.cs
public sealed class PlayerMover
{
}
잘못된 사용
// File: MovePlayer.cs
public sealed class MovePlayer
{
}

역할은 타입명에, 동작은 함수명에 둔다.

14

Unity 객체 이름과 스크립트 이름을 맞춘다

올바른 사용
Prefab: InventoryPanel
Script: InventoryPanel.cs
Class: InventoryPanel
잘못된 사용
Prefab: Inventory
Script: BagPanel.cs
Class: BagPanel

Inspector 이름과 코드 검색 이름이 같은 개념을 가리켜야 한다.

SECTION

단어

15

관용 다음절어는 한 단어처럼 취급한다

올바른 사용
Hotkey
Infobox
Inputbox
Lifetime
Nickname
Filename
잘못된 사용
HotKey
InfoBox
InputBox
LifeTime
NickName
FileName

플랫폼/서드파티 관용은 따르되 새 이름은 한 단어처럼 다룬다.

16

다른 키워드의 부분 문자열이면 피한다

올바른 사용
StageState
IntermissionState
잘못된 사용
MissionState
IntermissionState

Intermission 안에 Mission이 들어간다. 검색하면 Mission이 의도치 않게 같이 잡힌다.

17

복수 그룹은 현재 -s/-es를 허용한다

올바른 사용
Items
Enemies
Rewards
잘못된 사용
ItemList
EnemyArray
RewardBucket

List, Array는 헝가리언처럼 보일 수 있다. 더 나은 대안은 미정이다.

SECTION

구문과 구성

18

외부 생명주기 객체는 읽기 전용 속성으로 둔다

올바른 사용
public Player Player { get; }
잘못된 사용
public Player Player;

생성과 소유권이 외부에 있으면 setter를 닫는다.

19

var를 적극 사용한다

올바른 사용
var items = new Dictionary<string, Item>();
잘못된 사용
Dictionary<string, Item> items =
    new Dictionary<string, Item>();

오른쪽 표현식으로 타입을 알 수 있으면 var를 쓴다.

20

신규 using 형태는 사용하지 않는다

올바른 사용
using (var stream = File.OpenRead(path))
{
}
잘못된 사용
using var stream = File.OpenRead(path);

기존 using 블록 형태를 유지한다.

21

sealed는 override 앞에 둔다

올바른 사용
public sealed override void Dispose()
{
}
잘못된 사용
public override sealed void Dispose()
{
}

접근 제한자, sealed, override 순서로 읽히게 한다.

22

클래스 내부 섹션은 변수, 속성, 이벤트, 함수 순서로 둔다

올바른 사용
private enum StateEnum { }
private struct Entry { }
private interface CacheInterface { }
private sealed class Cache { }
private const int MaxCount = 10;
public int Count { get; }
public event Action Changed;
public void Refresh() { }
잘못된 사용
public void Refresh() { }
public int Count { get; }
private enum StateEnum { }

inner enum, inner struct, inner interface, inner class 다음은 변수 > 속성 > 이벤트 > 함수 순서다.

23

각 카테고리 내부는 abc순으로 둔다

올바른 사용
private int alphaCount;
private int betaCount;
private int rewardCount;

public bool IsActive { get; }
public int RewardCount { get; }
public string TooltipText { get; }

public event Action Changed;
public event Action Selected;

public void ApplyReward()
{
}

public void RefreshTooltip()
{
}
잘못된 사용
private int rewardCount;
private int alphaCount;
public void ApplyReward()
{
}
public bool IsActive { get; }
public event Action Selected;
public string TooltipText { get; }
public event Action Changed;
public void RefreshTooltip()
{
}

abc 정렬은 섹션 순서, 상속 구현 순서, 생성자 우선 규칙을 먼저 적용한 뒤 같은 상속 그룹의 같은 카테고리 내부에서만 적용한다. abc 정렬 때문에 부모/인터페이스 구현 그룹을 섞지 않는다.

24

상속 구현은 부모 선언 순서대로 둔다

올바른 사용
public sealed class RewardButton : ButtonBase, ClickableInterface, TooltipInterface
{
    private int buttonIndex;
    private int clickableCount;
    private int tooltipCount;
    private int rewardCount;

    public bool IsButtonActive { get; }
    public bool IsClickable { get; }
    public string TooltipText { get; }
    public int RewardCount { get; }

    public RewardButton()
    {
    }

    protected override void RefreshButton()
    {
    }

    void ClickableInterface.Click()
    {
    }

    void TooltipInterface.ShowTooltip()
    {
    }

    public void GiveReward()
    {
    }
}
잘못된 사용
public sealed class RewardButton : ButtonBase, TooltipInterface, ClickableInterface
{
    private int rewardCount;
    private int tooltipCount;
    private int buttonIndex;

    public int RewardCount { get; }
    public string TooltipText { get; }

    public void GiveReward()
    {
    }

    public void ShowTooltip()
    {
    }

    protected override void RefreshButton()
    {
    }

    public RewardButton()
    {
    }
}

부모 > 자식 순서다. 부모 간에는 상속 선언 순서를 따르고, 여러 인터페이스의 선언과 구현은 abc순으로 둔다. 생성자는 함수 중 가장 먼저 둔다.

25

인터페이스 구현은 타입 explicit으로 둔다

올바른 사용
public sealed class TooltipButton : TooltipInterface
{
    string TooltipInterface.TooltipText => tooltipText;

    void TooltipInterface.ShowTooltip()
    {
    }
}
잘못된 사용
public sealed class TooltipButton : TooltipInterface
{
    public string TooltipText => tooltipText;

    public void ShowTooltip()
    {
    }
}

인터페이스 멤버는 public으로 노출하지 않고 InterfaceName.MemberName 형태로 구현한다.

26

멤버 섹션은 const, readonly, static, member 순서로 둔다

올바른 사용
private const int MaxCount = 10;

private readonly int StartIndex;

private static int __cacheCount;
private static int __index;

private int count;
private int index;
잘못된 사용
private const int MaxCount = 10;
private int index;
private readonly int StartIndex;
private static int __cacheCount;
private int count;

const, readonly, static, member는 서로 다른 섹션이다. 각 섹션 안에서 abc순으로 둔다.

27

같은 제어문 연쇄에서는 중괄호 사용을 일관되게 맞춘다

올바른 사용
if (isReady)
    StartGame();
else if (canRetry)
    RetryGame();
else
    CancelGame();

if (isReady)
{
    StartGame();
}
else if (canRetry)
{
    ResetGame();
    RetryGame();
}
else
{
    CancelGame();
}
잘못된 사용
if (isReady)
    StartGame();
else if (canRetry)
{
    ResetGame();
    RetryGame();
}
else
    CancelGame();

모든 분기가 한 문장이면 중괄호를 생략하는 방식을 권장한다. 같은 if, else if, else 연쇄에서 한 분기라도 중괄호가 필요하면 나머지 분기에도 모두 중괄호를 사용한다.

28

증감 연산자는 전위형으로 쓴다

올바른 사용
++index;
--index;
잘못된 사용
index++;
index--;

증가와 감소 연산자는 각각 ++index, --index처럼 피연산자 앞에 둔다. index++, index-- 같은 후위형은 사용하지 않는다.

29

논리 부정 연산자는 Boolean 토글 대입에만 쓴다

올바른 사용
isActive = !isActive;

if (isActive == false)
    Activate();
잘못된 사용
if (!isActive)
    Activate();

return !isActive;

논리 부정 연산자 !는 Boolean 값을 토글하여 같은 변수에 다시 대입할 때만 사용한다. 조건문과 반환식을 포함한 그 밖의 모든 경우에는 if (isActive == false)처럼 == false를 사용해 명시적으로 비교한다.

30

접근 제한자에 따라 멤버를 정렬하지 않는다

올바른 사용
public int AlphaCount { get; }
private int BetaCount { get; }
protected int EnemyCount { get; }
private int ItemCount { get; }
public int RewardCount { get; }
protected int StageCount { get; }
잘못된 사용
public int AlphaCount { get; }
public int RewardCount { get; }
protected int EnemyCount { get; }
protected int StageCount { get; }
private int BetaCount { get; }
private int ItemCount { get; }

접근 제한자는 멤버 정렬 그룹을 만들지 않는다. public, protected, private끼리 모으지 말고 섹션 순서, 상속 구현 순서, 생성자 우선 규칙을 적용한 뒤 같은 상속 그룹의 같은 카테고리 안에서 이름순으로 정렬한다.

31

Attribute는 한 줄에 하나씩 명시한다

올바른 사용
[Header("Settings")]
[SerializeField]
private int count;
잘못된 사용
[Header("Settings"), SerializeField]
private int count;

[SerializeField] private int index;

각 Attribute는 한 줄에 하나씩 작성하고, Attribute가 적용되는 선언은 다음 줄에 둔다. 여러 Attribute를 한 줄에 합치거나 Attribute와 대상 선언을 같은 줄에 작성하지 않는다.

32

개별 조건식이 두 개 이상 연결되면 각 조건식을 괄호로 감싼다

올바른 사용
if ((count > 0) && (item != null))
    Use(item);

if ((isReady == true) || ((count > 0) && (item != null)))
    Start();
잘못된 사용
if (count > 0 && item != null)
    Use(item);

if (isReady == true || count > 0 && item != null)
    Start();

논리 연산자로 두 개 이상의 개별 조건식을 연결할 때는 연산자 우선순위와 관계없이 각 조건식을 괄호로 감싼다. 중첩된 복합 조건식은 바깥 단계에서 하나의 조건식으로 보고, 내부 조건식에도 같은 규칙을 재귀적으로 적용한다. 사람이 연산자 우선순위를 기억하고 있다는 전제보다 조건 경계를 코드에 명시하는 방식을 우선한다.
