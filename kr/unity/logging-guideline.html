공통 형식

  • 사용자에게 전달되거나 자동 비교하는 영어 로그의 고정 문구는 모두 대문자로 쓴다.
  • 실행 중 결정되는 모든 identifier는 보간식 {expression}으로 참조한다.
  • 일반 로그의 최종 출력에는 행위 주체자가 가장 앞에 {actor}> 형태로 있어야 한다. 태그 로거가 붙이면 호출 문자열에는 중복 작성하지 않는다. catch된 예외 객체를 그대로 출력하는 로그는 제외한다.
  • 문자열 연결보다 $"..."을 사용하고 identifier의 실제 값은 대문자로 바꾸지 않는다.
  • : 는 분류와 값, > 는 주체와 사건, -> 는 상태 변화를 구분한다.
  • catch된 예외는 주체자나 설명 문구를 붙이지 않고 e 또는 e.ToString()을 그대로 예외 API에 전달한다.
  • 민감한 인증정보와 불필요한 개인정보는 기록하지 않는다.
Debug.Log($"{name}> ENABLED.");
Debug.Log($"{GetType().Name}> APP STEP : {_step} -> {value}");
Debug.LogWarning($"{GetType().Name}> NO HANDLERS : {request.GetType().Name}");
Debug.LogException(e);

{}는 출력에 중괄호를 표시하는 규칙이 아니라 고정 문구와 identifier 참조를 소스에서 분리하는 규칙이다.

행위 주체자와 시스템 태그

실제로 저장·전송·표시되는 최종 로그의 첫 토큰만 보고도 “누가 이 일을 했는가?”를 알아야 한다. 메시지 속 대상 identifier는 행위 주체자를 대신하지 않는다.

  • 인스턴스는 {name}>, 서비스·컴포넌트는 {GetType().Name}>를 기본으로 한다.
  • 사용자 요청은 {memberId}:{GetType().Name}>, 네트워크 연결은 {connectionName}>처럼 컨텍스트를 조합할 수 있다.
  • 주체가 하나뿐이고 절대 모호하지 않은 전역 부팅 로그만 제한적으로 생략할 수 있다.
  • SEND>, RECV>, HANDLED>는 네트워크 의미를 내포한 축약형이지만, 동시 연결·요청이 섞이면 실제 연결 태그를 앞에 둔다.
  • 주체자를 정하기 어렵거나 기존 로그에 없다면 생략하지 말고 책임 객체와 계층 설계를 다시 검토한다.
  • catch 예외 로그는 예외다. Debug.LogException(e), logger.WriteExceptionLine(e.ToString()), Console.WriteLine(e)처럼 예외 자체만 전달한다. 기존 태그 로거가 자동으로 붙이는 태그는 그대로 둔다.

프레임워크에서 보장한다

개별 호출부가 매번 주체자를 기억하게 하지 않는다. 부모 클래스, 공통 로거, 미들웨어, 베이스 컨트롤러, 패킷 핸들러 팩터리가 태그를 한 번 받고 모든 하위 로그에 붙이는 방식을 우선한다. 태그가 설정된 로거의 호출 문자열은 사건만 쓰고 주체자를 중복 작성하지 않는다.

var tag = $"{memberId}:{GetType().Name}";
logger.Tag = tag;

logger.WriteNoticeLine("RECEIVED.");
logger.WriteNoticeLine("START HANDLE.");
logger.WriteNoticeLine($"END HANDLE : {elapsedMilliseconds} MS");
logger.WriteNoticeLine($"RESPONSE : {result} ({bytes.Length} BYTES)");

최종 출력

{memberId}:{controllerType}> RECEIVED.
{memberId}:{controllerType}> START HANDLE.
{memberId}:{controllerType}> END HANDLE : {elapsedMilliseconds} MS
{memberId}:{controllerType}> RESPONSE : {result} ({bytesLength} BYTES)

ASP.NET 요청 계층은 사용자·세션·컨트롤러 태그를 상위에서 구성하고 수신·처리·응답 로그를 공통 기록한다. 자식 계층은 태그가 부착된 로거를 전달받는다.

01 REQUIRED SYSTEM LOG

필수 시스템 로그

정상 실행에서 100% 항상 출력되는 기준점이다. 실행 환경, 주요 시스템의 시작과 완료, 초기화 순서, 상태 전이를 기록하고 이후 로그의 앞뒤와 누락 여부를 자동 비교할 수 있게 한다.

  • 빌드 플래그로 제거하지 않는다.
  • 같은 정상 실행에서는 같은 기준점이 같은 순서로 출력되어야 한다.
  • 주요 시스템의 시작과 완료를 짝으로 기록한다.
  • 부모나 공통 초기화 계층이 컴포넌트 태그를 부착한다.
  • 문구 변경·삭제·순서 변경은 로그 계약 변경으로 취급한다.
  • 반복되는 세부 스텝은 필수 로그가 아니라 디버깅 로그로 둔다.
Debug.Log($"{applicationName}> APPLICATION START : {Application.version} ({Application.platform})");
Debug.Log($"{system.GetType().Name}> INITIALIZE BEGIN.");
Debug.Log($"{system.GetType().Name}> INITIALIZE END.");
판단 질문: 이 줄이 없거나 순서가 달라졌을 때 정상 실행이 달라졌다고 판단할 수 있는가?
02 WARNING LOG

경고 로그

기대하는 환경·값·사용법에서 벗어났지만 실행은 계속할 수 있는 모든 상황을 기록한다. 정상 실행과 테스트에서는 경고 0개를 유지한다.

  • 복구했다는 이유로 이상 상태를 숨기지 않는다.
  • 잘못된 사용, 누락 설정, 범위 이탈, 예상하지 않은 null을 기록한다.
  • 경고에도 주체자를 붙여 어느 컴포넌트·요청·사용자가 벗어났는지 드러낸다.
  • 경고를 없앨 때 출력만 삭제하지 말고 원인을 수정한다.
  • 반복 경고는 최초 1회나 의미 있는 상태 변화 시점으로 제한한다.
  • 계속 실행할 수 없는 상태는 오류 또는 예외로 처리한다.
Debug.LogWarning($"{name}> UI EFFECT NOT SET : {child.name}");
Debug.LogWarning($"{GetType().Name}> OUT OF INDEX : {index} / {values.Length}");
Debug.LogWarning($"{name}> NOT INITIALIZED.");
판단 질문: 기대 상태에서 벗어났지만 실행은 계속할 수 있는가?
03 DEBUGGING LOG

디버깅 로그

인캡슐레이션된 시스템 내부의 실제 값과 세부 스텝을 직접 확인할 때 사용한다. 네트워크, 멀티스레드, 운영체제, 파일 시스템처럼 외부 상태와 타이밍의 영향을 받는 코드에서 특히 중요하다.

  • 기본 실행에서는 끄거나 필터링할 수 있어야 한다.
  • 스레드·요청·연결 identifier로 병렬 흐름을 묶는다.
  • 시작·대기·재개·완료·취소 단계를 구분한다.
  • 유저에게 공개하지 않는 내부 로그는 빠른 판독을 위해 한글로 쓸 수 있다.
  • 영어로 쓰면 대문자와 identifier 규칙을 그대로 적용한다.
  • 네트워크 축약 동사도 동시 흐름이 섞이면 연결·요청 태그를 앞에 둔다.
logger.LogDebug($"{connectionName}> SEND : {request.GetType().Name} ({requestId})");
logger.LogDebug($"{threadId}> 응답 대기를 시작함 : {requestId}");
logger.LogDebug($"{connectionName}> RECV : {response.GetType().Name} ({requestId})");
판단 질문: 평상시 계약에는 필요 없지만 지금 내부 스텝이나 실제 값을 눈으로 확인해야 하는가?
OPTIONAL UX LOG

UX LOG

사용자 행동을 장기 집계해 UX 결정을 내릴 때 쓰는 선택적 분석 채널이다. 필수 로그나 경고를 대신하지 않는다.

  • 분석 목적이 있는 행동과 결과만 기록한다.
  • 이벤트 이름과 속성 스키마를 먼저 정하고 안정적으로 유지한다.
  • 사용자·세션·버전 등 공통 컨텍스트는 로거가 일관되게 부착한다.
  • 행위 주체자는 문구와 구조화된 사용자·세션 필드 양쪽에 보존한다.
  • 집계할 값은 자유 문장뿐 아니라 구조화된 속성으로 전송한다.
  • UX LOG 실패가 게임 진행을 막지 않게 분리한다.
  • 개인정보와 자유 입력 원문은 수집 목적과 보관 정책 없이 기록하지 않는다.
analyticsLogger.WriteLine(
    $"{user.Id}> USER LOGIN : {user.Nickname}",
    () => (nameof(AnalyticsEventGroup.User), new
    {
        UserId = user.Id,
        user.Nickname,
    }));
판단 질문: 이 기록으로 사용자 행동이나 UX 결과에 관한 어떤 결정을 내릴 것인가?

선택 순서

  1. 항상 존재하고 실행 순서 비교의 기준점이면 필수 시스템 로그다.
  2. 기대 상태에서 벗어났지만 계속 실행할 수 있으면 경고 로그다.
  3. 내부 구현의 세부 스텝을 확인하려면 디버깅 로그다.
  4. 사용자 행동을 집계해 UX 결정을 내리려면 UX LOG다.

한 사건이 여러 목적을 가지면 채널을 분리한다. 로그인 시스템 초기화는 필수 로그, 비정상 설정은 경고, 패킷 왕복은 디버깅 로그, 로그인 방식 선택은 UX LOG다.

검토 체크리스트

  • 로그 종류를 먼저 결정했는가?
  • 필수 로그의 기준점과 순서가 안정적인가?
  • 정상 상태에서 경고가 0개인가?
  • 디버깅 로그를 필터링할 수 있는가?
  • UX LOG에 분석 목적과 구조화된 속성이 있는가?
  • 외부 전달 영어 문구가 모두 대문자인가?
  • 모든 identifier가 {expression}으로 참조되는가?
  • 일반 로그의 최종 출력 첫 토큰에서 행위 주체자를 식별할 수 있는가?
  • 부모·공통 로거가 태그를 시스템적으로 보장하는가?
  • 태그 로거의 호출 문자열에 주체자를 중복 작성하지 않았는가?
  • 주체자 없는 전역·네트워크 축약 로그가 정말 모호하지 않은가?
  • catch 예외 로그에 주체자나 설명 문구를 붙이지 않고 예외 자체를 그대로 전달했는가?
  • 민감한 값이 없는가?
