Codex 공통 작업 지침

여러 프로젝트에서 반복되는 작업 운영 규칙을 한곳에 모은 기준 문서. 프로젝트별 기획과 아트 방향은 각 저장소의 Design.html에 두고, 공통 작업 방식은 이 문서를 우선 참조한다.

Home
Purpose
  • 개인 맞춤 설정에는 짧은 부트스트랩만 남기고, 반복되는 작업 규칙은 이 문서로 모은다.
  • 다른 PC나 새 Codex 세션에서도 같은 기준을 참조할 수 있도록 GitHub Pages URL을 고정한다.
  • 비밀 정보, 토큰, 개인 계정 식별자, 내부 서버 주소는 이 문서에 기록하지 않는다.
시작 순서 승인 경계 파일과 Git 문서 책임 검증 아트와 UI 보고
01

시작 순서

  1. 사용자의 최신 지시와 현재 대화의 작업 범위를 우선한다.
  2. Design.html이 있으면 작업 전에 읽고, 상단의 공통 작업 지침 링크를 함께 확인한다.
  3. README.md, 빌드 문서, 프레임워크 문서처럼 실제 실행에 필요한 문서를 확인한다.
  4. 작업 전 git status --short --branch로 더티 워크트리와 현재 브랜치를 확인한다.
02

승인 경계

되돌리기 어렵거나 외부 상태를 바꾸는 작업은 직전 대화에서 명시적으로 지시된 경우에만 바로 진행한다.

즉시 수행

맞춤형 지침에 명시된 일반 작업, 코드 수정, 문서 초안 작성, 로컬 검증은 별도 승인 없이 진행한다.

직전 명령 필요

git push, 공개 배포, 완전 삭제, 강제 되돌리기, 이전 상태로 복구 불가한 문서 정리는 직전 지시가 없으면 확인받는다.

리뷰 모드 존중

사용자가 리뷰, 비교, 설명을 요청한 경우 먼저 분석과 근거를 제공하고, 구현은 별도 지시가 있을 때 시작한다.

새 지시 우선

작업 중 사용자가 새 메시지를 보내면 최신 지시가 기존 계획보다 우선한다.

03

파일과 Git

인코딩과 줄바꿈

  • 기존 텍스트 파일은 원본 인코딩과 줄바꿈을 유지한다.
  • 원본이 없는 새 텍스트 파일은 UTF-8 No-BOM과 CRLF를 기본으로 한다.
  • Windows PowerShell 출력이 한글을 잘못 표시할 수 있으므로, 필요하면 바이트 단위 또는 UTF-8 명시 읽기로 확인한다.

더티 워크트리

  • 사용자나 다른 스레드가 만든 변경은 의도된 작업으로 본다.
  • 요청받지 않은 변경을 되돌리거나 스타일 정리하지 않는다.
  • 같은 파일을 만져야 하면 요청된 최소 범위만 수정하고 기존 변경을 보존한다.

스테이징과 커밋

  • git add -A로 넓게 담지 말고, 작업 소유 파일만 좁게 스테이징한다.
  • 커밋 전 git diff --cached --name-status와 필요한 경우 git diff --cached로 범위를 확인한다.
  • 여러 저장소를 동시에 다룰 때는 저장소별로 독립 커밋한다.
  • Design.html과 DesignAssets는 기획 문서 작업이 명시된 경우에만 커밋에 포함한다.

GitHub Projects 상태

  • GitHub 원격 저장소가 있는 프로젝트는 origin에서 OWNER/REPO를 확인하고, gh repo view OWNER/REPO --json projectsV2로 저장소에 연결된 Project를 확인한다.
  • 연결된 Project가 있으면 태스크 시작, 진행, 완료 흐름에 맞춰 대응되는 issue, pull request, 또는 project item의 상태를 함께 갱신한다.
  • 연결된 Project가 없거나 대응 item이 명확하지 않으면 임의로 만들거나 옮기지 말고, 작업 보고에 생략 사유나 확인 필요 사항을 남긴다.
04

문서 책임

공통 작업 지침

반복되는 작업 운영 규칙, 승인 경계, Git 규칙, 검증 원칙을 이 저장소에서 직접 관리한다.

Design.html

프로젝트의 목표, 기획 의도, 아트 스타일, UX 기준, 결정 사유를 둔다.

코딩 컨벤션

언어별 스타일은 별도 컨벤션 문서로 분리하고, 프로젝트 문서는 필요한 링크만 둔다.

기획 문서 작성 기준

  • Design.html은 대화 기록이 아니라 최종 결정과 이유를 보존하는 문서다.
  • 구현 이후 코드와 기획서가 다르면, 최신 코드와 사용자 의도를 기준으로 기획서를 갱신한다.
  • 중간 후보, 폐기안, 비교용 임시 파일은 최종 문서가 참조하지 않는 한 문서 자산 폴더에 남기지 않는다.
05

검증

  • 가능한 검증은 실제 명령으로 실행하고 결과를 보고한다.
  • 일반 텍스트와 문서 변경은 git diff --check, 인코딩, 줄바꿈, 후행 공백을 확인한다.
  • HTML 문서는 주요 앵커, 링크, 이미지 경로, 큐 마커, 접힌 섹션 수처럼 깨지기 쉬운 표면을 확인한다.
  • Unity 작업은 관련 .meta GUID, prefab YAML 참조, 빌드 결과, 에디터 로그 등 실제 증거를 우선한다.
  • 검증을 실행하지 못한 경우 이유를 숨기지 않고 짧게 남긴다.
06

아트와 UI

  • 새 래스터 아트, UI 아트, 게임 에셋, 아이콘형 이미지, 시각 레퍼런스의 첫 생성과 첫 미감 결정은 반드시 이미지젠/imagegen 스킬을 기본으로 한다.
  • UI처럼 규격에 민감한 작업도 예외가 아니다. 먼저 이미지젠으로 아트 스타일, 실루엣, 재질감, 분위기, 전체 품질을 확정한 뒤 UI 규격화로 넘어간다.
  • System.Drawing, PIL, ImageMagick, SVG, 캔버스, HTML/CSS, 절차적 드로잉 코드는 첫 이미지 생성 도구가 아니라 후처리 도구다. 이미지젠 결과가 승인된 뒤에만 리사이즈, 크롭, 패딩, 슬라이싱, 알파 정리, 포맷 변환, 상태별 파생, 나인슬라이스 가이드, 아틀라스 패킹 같은 보강에 사용한다.
  • 이미지젠 결과의 아트 스타일이 틀렸다면 코드 후처리로 구제하려 하지 말고 이미지젠으로 다시 생성한다.
  • 최종 또는 현재 문서가 참조하는 에셋만 DesignAssets에 둔다.
  • 실제 런타임 에셋은 문서용으로 복제하지 않고 실제 프로젝트 경로를 직접 참조한다.
07

보고

  • 피드백과 문서 작성은 기본적으로 한글로 한다.
  • 최종 보고는 바뀐 파일, 핵심 변경, 검증 결과, 남은 위험을 짧게 정리한다.
  • 리뷰 요청의 답변은 발견한 문제를 먼저 제시하고, 요약은 뒤에 둔다.
  • 명령 출력 전체를 사용자가 볼 수 없으므로, 중요한 결과는 답변에 직접 전달한다.
