역할 분리

.codexCodex 실행 규칙

작업 운영 규칙은 GitHub의 Codex 공통 작업 지침을 참조합니다. 기획서 작성 규칙은 이 문서와 project-design-document-router 스킬이 가리키는 공개 기준을 따릅니다.

Review눈으로 보는 검토본

이 HTML은 사용자가 브라우저에서 규칙, 표본, 검증 기준을 확인하기 위한 공개 문서입니다. 로컬 MD는 Codex 내부 참고용이며, 사용자가 직접 확인할 기준은 GitHub Pages의 이 HTML입니다.

프로젝트 안에 README.md나 AGENTS.md가 항상 있다고 가정하지 않습니다. 현재 기준의 중심은 프로젝트의 Design.html과 GitHub의 공통 작업 지침입니다.

영역 분리: 작업 지침과 기획서 작성 지침은 서로 다른 문서 영역입니다. 공통 작업 지침은 Codex가 어떻게 작업하고 검증하고 커밋 범위를 지킬지 다루며, Design.html은 게임/제품의 현재 결정, 구현 기준, UX 사유만 다룹니다. 작업 절차, 큐 운영, 검증 명령, 커밋 규칙은 Design.html 본문에 넣지 않습니다.

작업 시작 기준

  1. 프로젝트의 live Design.html을 다시 열고, 작업 운영은 GitHub의 공통 작업 지침을 확인합니다.
  2. README.md는 보통 GitHub 공개용 문서로 취급합니다. 프로젝트가 명시하지 않는 한 작업 지침의 원본으로 보지 않습니다.
  3. AGENTS.md가 있으면 읽되, 새 프로젝트에 반드시 있어야 한다고 전제하지 않습니다.
  4. git status --short --branch로 기존 변경과 병렬 작업 흔적을 확인합니다.
  5. 현재 프로젝트 파일을 진실 기준으로 삼고, 오래된 메모리나 외부 초안만으로 판단하지 않습니다.

중복 보존 기준

개인별 Codex 설정은 중요하지만 여러 환경에서 동기화되지 않을 수 있습니다. 그래서 모든 개인 설정을 복사하지는 않되, 빠지면 손상이 생길 수 있는 규칙은 프로젝트 문서에 반복해서 남깁니다.

  • 되돌리기 어려운 작업은 직전 사용자 지시가 없으면 승인받습니다.
  • 원본 인코딩과 줄바꿈을 보존하고, 새 텍스트는 UTF-8 No-BOM과 CRLF를 기본으로 합니다.
  • 피드백과 문서 작성은 한국어를 우선합니다.
  • Design.html이 있으면 작업 전에 읽고, 기획 상태가 바뀌면 갱신합니다. 작업 운영 규칙은 프로젝트 문서가 아니라 공통 작업 지침 URL을 기준으로 확인합니다.
  • 마지막 기획 결과물 이후 코드가 직접 수정되어 Design.html과 다르면, 코드가 구현 기준입니다. 기획서는 코드와 최신 의도에 맞춰 갱신합니다.
  • 관련 없는 dirty 변경은 되돌리지 않고 유지합니다.

Design.html 생성 기준

Design.html은 사람도 읽고 Codex도 읽는 게임 내용 지침입니다. 무엇을 만들지뿐 아니라 왜 그렇게 정했는지를 보존해서, 나중 작업이 방향을 잃지 않게 합니다.

기획서는 전체 대화나 모든 중간 결과를 보관하는 히스토리 문서가 아닙니다. 최종적으로 결정된 UX와 구현 기준, 현재 구현 상태, 그 결정을 택한 사유만 남깁니다.

  • 다음 버전이나 최종 결과물이 나오면 임시 비교안, 폐기된 후보, 중간 산출물 설명을 제거합니다.
  • 기획서 작성 중 생긴 후보 이미지·영상·소리, 비교표, 접촉시트, 임시 변환물, 폐기 초안은 프로젝트 루트 $Trash 아래에만 둡니다. DesignAssets/GuidelinesAssets에는 현재 최종 문서가 실제 참조하는 확정 자료만 둡니다.
  • 현재 구현과 100% 일치하는 기준 문서를 목표로 하며, 구현되지 않은 가정은 미해결 항목으로 분리합니다.
  • 코드와 기획서가 충돌할 때, 코드가 더 최신 구현을 담고 있으면 코드를 우선하고 기획서를 코드에 맞춰 정리합니다.
  • 글자 설명만 있는 기획서는 아직 완성본으로 보지 않습니다. 모든 기획 내용은 최종적으로 에셋, 화면, 동작, 사운드, 데이터 구조로 구현되어야 하므로 문서도 가능한 한 그 결과물의 형태로 보여줍니다.
  • 이미지, 영상, 소리, 미리보기, 도형, 흐름도, 상태 전환 다이어그램처럼 바로 이해되는 자료를 본문 설명보다 우선합니다. 동작 설명도 긴 문단 대신 도형, 단계 그림, 상태표, 짧은 캡션으로 표현합니다.
  • 글은 자료만으로 부족한 부분과 “왜 이 결정을 했는가”를 설명하는 데만 씁니다. 반드시 글이 필요할 때도 가장 짧게 쓰고, 사용자의 긴 설명을 그대로 옮기지 않습니다.
  • 도메인 한정 용어, 고유 명사, 프로젝트 내부 이름, 약어, 아트/시스템 용어처럼 추가 설명이 필요한 항목은 문서 표면에는 물음표 아이콘만 두고, 마우스오버나 포커스 때 프로젝트 안에서의 의미를 짧게 보여줍니다. 툴팁을 적극 사용합니다.
  • Codex가 용어의 프로젝트 내 의미를 모르면 임의로 정의하지 않고 사용자에게 물은 뒤 그 답을 툴팁에 반영합니다.
  • 기획서 작성 규칙 자체는 이 리뷰 문서와 스킬에 두고, 실제 Design.html 본문에는 장황한 메타 규칙을 노출하지 않습니다.
  • 섹션 링크는 본문 상단 카드가 아니라 왼쪽 floating 목차로 분리해, 긴 문서에서도 콘텐츠와 내비게이션이 서로 섞이지 않게 합니다.
  • 작성 완료됐거나 현재 구현·결정이 들어 있는 섹션을 본문 앞쪽에 우선 배치합니다. 사용자는 먼저 읽을 가치가 있는 확정 내용을 보아야 합니다.
  • 미작성 섹션 후보가 많을 때는 본문 중간에 우다다다 늘어놓지 않습니다. 뒤쪽에 미작성 섹션 또는 추후 작성 후보 같은 큰 접힘 섹션 하나로 모아 닫아두고, 그 안에서만 범주별 목록과 작성 필요 태그를 표시합니다.
  • floating 목차도 작성 완료 섹션을 먼저 보여주고, 미작성 묶음은 뒤쪽의 단일 링크나 접힘 그룹으로 둡니다. 필요하면 그룹 제목에 작성 필요 N처럼 남은 작성 상태를 표시합니다.
  • 새 기획서의 첫 구조는 기본 범주를 후보로 삼되, 비어 있는 후보는 앞쪽 본문을 밀어내지 않게 뒤쪽 접힘 묶음에 둡니다. 사용자가 직접 열기 전까지 미작성 후보가 읽기 흐름을 방해하지 않아야 합니다.
  • 범주별 구현 기준을 별도 섹션으로 모아두지 않습니다. 범주 기준과 결정 사유는 각 섹션 안에서 해당 데이터, 이미지, 표, 사운드와 붙여 둡니다.
  • 헤더 바로 아래에 문서 신호, 작성 지침, 범주 요약 같은 메타 섹션을 두지 않습니다. 첫 본문 섹션부터 실제 게임/제품 내용을 다룹니다.
  • Design.html 헤더에는 문서 제목과 외부 참조 링크 두 개만 둡니다. 링크는 Codex 공통 작업 지침, 기획서 작성 기준을 가리키되, 링크 자체는 참조용 내비게이션일 뿐 Design.html 본문에 작업 지침 내용을 섞는 근거가 아닙니다.
  • Design.html 헤더는 스크롤 중에도 상단에 유지되는 sticky 문서 크롬으로 둡니다. 헤더가 본문을 가리지 않도록 좌측 floating 목차의 top offset과 섹션 scroll-margin을 헤더 높이에 맞춰 조정합니다. 좌측 목차는 본문 바깥 여백이 충분한 넓은 화면에서만 표시하고, 좁은 화면에서는 숨겨 본문과 겹치지 않게 합니다.
  • 현재 레이아웃 표본은 H:\Mines\Design.html입니다. 새 기획서나 기존 기획서 갱신은 Mines의 구조처럼 sticky 헤더, 좌측 floating 목차, 작성 완료 섹션 우선 배치, 뒤쪽 미작성 후보 접힘 묶음을 기본 골격으로 삼습니다.

게임 기획 기본 범주

첫 Design.html을 만들 때는 아래 범주를 기본 얼개로 삼고, 프로젝트에 맞지 않는 항목은 삭제하거나 합칩니다. 퍼즐 스테이지, 공장 물류, 전략 외교 같은 장르 전용 항목은 처음부터 독립 대분류로 만들지 말고 가까운 일반 범주에 넣습니다.

  • 핵심 방향: 정체성, 장르, 진행 방식, 핵심 재미, 승패, 난이도, 숙련
  • 세계와 공간: 세계관, 스토리, 세력, 월드/지역/던전, 배경, 이동과 발견
  • 레벨과 콘텐츠 구조: 스테이지, 챕터, 미션, 캠페인, 난이도 곡선, 콘텐츠 주기
  • 캐릭터와 성장: 캐릭터, 역할, 스탯, 스킬, 특성, 상태이상, 레벨, 동료와 적
  • 규칙과 계산 모델: 턴/틱, 공식, 점수, 확률, 생성 규칙, AI, 경제/전투 해석
  • 아이템과 경제: 아이템, 장비, 자원, 재화, 보상, 드롭, 인벤토리, 상점, 도안, 제작
  • 퀘스트와 이벤트: 퀘스트, 의뢰, 이벤트, 업적, 랭킹 목표, 튜토리얼, 반복/엔드 콘텐츠
  • 상호작용과 전투: 조작, 타겟팅, 보드/타일 규칙, 액션 피드백, 실패와 재도전
  • 연출과 커뮤니케이션: 다이얼로그, 보이스, 컷신, 카메라, 사운드, UI 문구, 알림과 로그
  • UI와 편의 시스템: HUD, 메뉴, 설정창, 저장, 접근성, 입력, 현지화, 도움말과 툴팁
  • 메타 시스템: 해금, 컬렉션, 진행도, 멀티플레이, 로비/매칭, 소셜, 서버 동기화, 라이브 운영
  • 미결정 / 제외 기준: 열린 질문, 보류 항목, 제외 방향, 리스크, 다음 버전 후보

기획 위계

Design.html은 위에서 아래로 내려갈수록 가장 큰 얼개에서 구체적인 디테일로 이동합니다. 첫 섹션은 펼쳐서 목표, 장르, 진행방식 같은 큰 답을 보여주고, 아래 섹션들은 같은 추상화 레벨끼리 묶어 접힌 상태로 둡니다.

  • 이 게임은 무엇이 목표인가: 목표, 장르, 진행방식, 표면 약속
  • 어떻게 진행되는가: 진행 루프, 재미를 느끼는 지점, 잘하게 되는 방법
  • 기획자는 무엇을 의도하는가: 유저가 배우거나 오해하거나 나중에 발견해야 하는 것
  • 그 다음에 세계, 시스템, 콘텐츠, 표현, 예외, 미정, 변경 기록으로 내려갑니다.

특히 잘 적어야 하는 것

  • 이 기능이나 시스템이 존재하는 이유
  • 다른 선택지 대신 이 방향을 고른 이유
  • 그 결정이 해결한 문제
  • 결정을 제한한 기술, UX, 세계관, 제작 범위 조건
  • 후속 작업자가 유지해야 하는 설계 의도

기본 포함 항목

  • 현재 게임/제품 방향
  • 사용자 경험 목표
  • 핵심 시스템과 기능 영역
  • 화면, 흐름, 콘텐츠, 규칙 계획
  • 결정 이유가 포함된 결정 기록
  • 선택된 방향을 설명하는 데 도움이 되는 기각된 대안
  • 미해결 질문과 위험
  • 에셋과 참고 자료 메모
  • 변경 기록

작성자용 운영 규칙은 공통 작업 지침 문서에 두고, Design.html은 게임 내용과 결정 이유를 중심으로 유지합니다. 기획서 작성 규칙은 문서의 구조와 표현을 정하는 외부 기준이며, 실제 기획서 본문에 작업 운영 규칙처럼 복사하지 않습니다.

표본 파일

표본은 고정된 프로젝트가 아니라 가장 최근에 공유 규칙을 적용해 편집·검증한 live 문서로 갱신됩니다. ImageStudio와 Overlord는 역사적 seed이며, 모든 문서에 규칙이 전파된 뒤에는 더 최신 편집본이 다음 표본이 됩니다.

Design롤링 표본

새 Design.html 작업은 현재 프로젝트 문서를 먼저 보고, 그 다음 최근 편집·검증된 live Design.html을 기준으로 삼습니다.

현재 live 표본: H:\Mines\Design.html. 역사적 seed: H:\Overlord\Design.html. 스냅샷: samples\Overlord.Design.sample.html
